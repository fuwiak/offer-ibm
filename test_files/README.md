# Golden set — эталонные заявки для оценки экстракции

Это набор реальных заявок (документов от клиентов), по которым проверяется
качество пайплайна `parseInquiryText` (`server/utils/offerKp/parseInquiry.js`):
правильно ли вырезаются позиции, единицы измерения и количества.

Тест-раннер: `server/__tests__/utils/offerKp/goldenSet.test.js`.
Он сам находит все папки-примеры внутри `test_files/` — новый пример не
требует изменений в коде теста.

## Что добавить, чтобы завести новый эталонный пример

Каждый пример — отдельная папка `test_files/<Name>/`. Внутри неё:

```
test_files/
  <Name>/
    <Name>.pdf              # исходный документ клиента
    <Name>.expected.csv     # эталон извлечения
```

плюс один файл вне этой папки (см. пункт 2 ниже) — фикстура с текстом
таблицы для юнит-теста.

1. **`test_files/<Name>/<Name>.pdf`** (или `.docx`/`.xlsx` — исходный
   формат клиента)
   Оригинальный документ как есть. Хранится для истории/повторной
   генерации фикстуры ниже, тестом напрямую не читается (OCR/ingest в
   юнит-тестах не гоняется — он завязан на внешние сервисы).

2. **`server/__tests__/fixtures/offerKp/<slug>-table.txt`**
   Текст таблицы **после** ingest/OCR — то, что реально прилетает в
   `parseInquiryText` в проде. Получить его можно, прогнав `<Name>.pdf`
   через обычный ingest приложения (или через ручной OCR) и скопировав
   результирующий текст таблицы как есть.

   `slug` = `<Name>` в нижнем регистре с `_` → `-`.
   Пример: `Slozhnost_vysokaya_1` → `slozhnost-vysokaya-1-table.txt`.

3. **`test_files/<Name>/<Name>.expected.csv`** — эталон, что должно
   получиться:

   ```csv
   nr,source_name,unit,quantity
   1,"Болт М10х100 (S16) ГОСТ 7805-70 ...","кг",30
   2,"Болт М10х20 (S16) ГОСТ 7805-70 ...","кг",14
   ```

   - `nr` — порядковый номер позиции (1-based, по порядку в документе).
   - `source_name` — **полное** наименование позиции, как оно должно быть
     извлечено (без домысливания, без обрезки хвостов вроде «30 кг»).
   - `unit` — единица измерения ровно в том виде, в котором её должен
     вернуть парсер (`кг`, `шт`, ...).
   - `quantity` — количество, целое число.

   Строк должно быть ровно столько, сколько реальных позиций в документе.

После этого просто запусти тесты — раннер сам подхватит новую пару файлов:

```bash
yarn test:golden
```

## Что проверяется и что репортится

`goldenSet.test.js` для каждого найденного примера:

- падает построчно (`it.each`), если наименование/ед.изм./количество не
  совпадает с эталоном 1-в-1 — это регрессионный тест, ловит поломки от
  правок промпта/парсера сразу;
- по завершении всех файлов печатает сводку в stdout:
  - **% позиций, извлечённых корректно** (наименование + ед.изм. + кол-во
    совпали полностью),
  - **% строк с корректными кол-во/ед.изм.** (даже если наименование
    отличается) — отдельно, т.к. это самая частая причина ошибок в КП,
  - число позиций по каждому файлу и суммарно.

Гоняй `yarn test:golden` после **каждого** изменения промпта/модели
экстракции и сравнивай цифры с предыдущим прогоном — без этого правки
промпта — это блуждание вслепую.

## Чего здесь пока нет: % корректных дополнений (matching)

Третья метрика из требования — «% корректных сопоставлений с каталогом»
(`matchInquiryLine` / `matchInquiryToDraft` в
`server/utils/offerKp/matchInquiryLines.js`) — **не автоматизирована**,
потому что matching идёт через живой поиск по каталогу (ShopDB +
LLM-агент), который не мокается в юнит-тестах.

Когда будет решение, как фикстурить каталог для тестов, эталон можно
расширить опциональными колонками:

```csv
nr,source_name,unit,quantity,matched_sku,matched_name,match_type
```

где `match_type` — `exact` / `analog` / `none`, как в
`matchInquiryLine`. До этого момента лучший источник golden-данных для
matching — лог корректировок оператора (система предложила → оператор
поправил): как только это логирование появится, из него можно собрать
реальные пары для тюнинга и regression-теста matching отдельно от
экстракции.
