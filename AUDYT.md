## 1. Скорость web UI

### Внедрённые правки ✅

| # | Изменение | Файл | Эффект |
|---|-----------|------|--------|
| 1 | Вырезание 20 неиспользуемых языков из entry bundle (UI поддерживает ru/pl/de/fr/kk + en) | `frontend/src/locales/resources.js` | ~1,5 MB меньше исходников в главном bundle (~200–400 KB меньше после минификации, меньше JS на парсинг при старте) |
| 2 | Заголовки `Cache-Control`: хэшированные чанки → `immutable, 1y`; `index.js`/`index.css`/HTML → `no-cache` (ETag/304); остальное → 1 день | `server/index.js` | Повторные визиты не качают чанки заново; entry валидируется через 304 |
| 3 | Дешёвый comparator `memo` — `JSON.stringify` больших RAG-источников считался для каждого сообщения на каждом chunk стрима | `frontend/.../HistoricalMessage/index.jsx` | Плавнее стриминг в длинных диалогах |

**Нужен rebuild фронтенда (`yarn prod:frontend`) и deploy, чтобы правки 1 и 3 попали на прод.**

### Что сделать (по убыванию приоритета)

1. **HTTPS/HTTP2.** Сайт ходит по `http://`. Помимо безопасности: нет HTTP/2 (медленнее параллельная загрузка чанков), нет PWA, нет clipboard API. Railway даёт TLS из коробки — настройте домен с сертификатом. *Самый большой одиночный выигрыш.*
2. **Brotli.** `compression()` в Node — только gzip. Entry 1,1 MB → gzip ~330 KB, brotli ~260 KB. Проще всего: CDN/nginx перед Node или прекомпрессия в build (`vite-plugin-compression`) + раздача `.br`.
3. **Entry bundle всё ещё ~1,1 MB.** После вырезки locales станет легче, но в entry статически импортируется всё из `main.jsx`. После сборки откройте `frontend/bundleinspector.html` и вырежьте следующих кандидатов (напр. `moment` — в 9 файлах, замена на `dayjs` или нативный `Intl` ≈ −60 KB; KaTeX/highlight в markdown — lazy).
4. **Ассеты без хэшей.** `assetFileNames` в `vite.config.js` возвращает исходные имена (шрифты, картинки) — поэтому у них только 1-дневный cache. Верните хэширование по умолчанию — тогда попадёт под `immutable`.
5. **Двойной scroll-handler** в `ChatHistory` (`onScroll={handleScroll}` + отдельный listener с `debounce`, создаваемый на каждый render) — мелочь, убрать.

## 2. UI/UX

### Проблемы

1. **Нет обратной связи на самом долгом этапе.** Enrich (parse PDF → matching строк → search agent) может идти десятки секунд, а первый сигнал пользователю (`offerKpQuotePanel`, `progressStage: "matched"`) приходит только ПОСЛЕ сопоставления. До этого — спиннер. **Рекомендация:** эмитить этапы раньше — `parsing` (после чтения PDF, с числом распознанных позиций), `searching` (старт matching, напр. счётчик «строка 12/40»). Бэкенд уже имеет SSE-канал и тип события — в основном дописать 2 эмиссии в `enrichInquiryLinesFromPdf` + обработку состояний в панели.
2. **Спиннер после конца ответа.** После последнего токена сервер ещё ждёт: post-processing → генерацию артефактов PDF/DOCX → **LLM follow-up suggestions** — и только потом шлёт `finalizeResponseStream`. Ответ выглядит «зависшим». **Рекомендация:** финализировать стрим сразу после артефактов, follow-up слать отдельным событием после факта (не блокируют запись чата).
3. **Meta/og-теги.** В `MetaGenerator` стоит `og:url: https://offer-kp.local` (placeholder) — превью ссылки в мессенджерах будет сломано. Dev `index.html` всё ещё брендирован «AnythingLLM».
4. **Auto-scroll при стриме** опирается на порог 2 px от низа — при быстром стриме с картинками/таблицами может «убегать». Наблюдать после внедрения progress-stage.

### Что хорошо

Lazy routing на уровне роутера, панель черновика КП пушится во время стрима, карточки файлов (Preview/Download) в чате, языковые версии с миграцией на ru.

## 3. Качество генерации и агента

### Что хорошо (не ломать)

- **Режим ShopDB-only для КП** — при запросе КП отключает vector search и web enrich, закрывая путь галлюцинаций цен. Триггер срабатывает и от одного прикреплённого PDF (`parsedTextHasQuoteSignals`), не только от фразы.
- **Правило цен:** только `exact`/`analog` получают цену из каталога; `similar`/`size_mismatch` → «под заказ» без подстановки чужой цены; подсказка похожего товара без его цены в строке КП. Это корректный дизайн.
- Per-line matching с fallback (search agent → ShopDB SQL agent), ошибка одной строки не валит остальные (`buildLineMatchErrorFallback`).
- Style-polish **пропускается** для КП (второй проход LLM не переписывает числа) — правильно.
- Отдельный каталожный промпт с жёсткими запретами («не пиши, что нет доступа к базе»).

### Риски / правки

1. **Латентность enrich = главное узкое место.** `SHOP_DB_ENRICH_TIMEOUT_MS` по умолчанию 60 с **+ полный retry после timeout** → до 120 с до первого токена. На большой заявке: N строк × (SQL search + возможный LLM fallback `searchAgent`) при `OFFER_KP_MATCH_CONCURRENCY=4`. **Рекомендации:** (a) снизить дефолтный timeout до ~30 с, retry только для упавшей фазы, не всего enrich; (b) поднять concurrency для запросов >20 строк (SQL выдержит); (c) кэш результатов matching строк per (текст строки → productId) в рамках треда — follow-up «добавь позицию 5» не должны пересчитывать всё с нуля.
2. **Двойные токены из PDF.** Полный текст вложения идёт в `contextTexts` («ПРИКРЕПЛЁННЫЙ ДОКУМЕНТ») *и параллельно* его позиции возвращаются блоками `[Каталог · PDF]`. В режиме КП с готовым draft сократите сырой текст PDF (напр. шапка + подвал + нераспознанные позиции) — меньше промпт, меньше стоимость, меньше конфликтов источников.
3. **Хрупкий re-parsing своего формата.** `autoQuoteArtifacts.parseCatalogBlock` вытаскивает цену/URL regex'ами из текста блока каталога, который сам сгенерировал `buildProductExcerpt`. Смена формата блока тихо сломает артефакты. **Рекомендация:** передавать структурные данные (объект товара) рядом с текстом блока вместо парсинга строки.
4. **Доп. вызовы LLM на request:** `quoteIntentJudge`, `searchAgent` fallback (на строку!), follow-up suggestions, OCR vision. Каждый — стоимость и латентность. Стоит добавить сводный счётчик в `metrics`/`ragTrace` (сколько sub-вызовов LLM на ответ), чтобы видеть регрессии.
5. **Тесты качества matching.** Есть harness (`test_files/*.expected.csv`) — расширьте кейсами: аналоги DIN↔ГОСТ, размер без покрытия, позиции вне каталога, PDF после OCR с ошибками. Это самый дешёвый способ держать качество КП при смене промптов.

## 5. Дополнение (2026-07-21): сильные/слабые стороны бэкенда, техники ускорения, оценка вакансии ML

Область расширена по запросу: полный аудит трёх подсистем (чтение файлов/OCR, matching ShopDB, генерация LLM) + внешний research (web search) по техникам ускорения + оценка, помогут ли компетенции из описания вакансии ML Engineer (BERT/LoRA/PEFT, matching/retrieval, quantization) этому проекту.

### 5.1 Сильные стороны (не ломать)

| Область | Что хорошо |
|---|---|
| OCR (`collector/utils/OCRLoader`, `SmartOCRAgent`) | 2-страничный «probe» перед полным OCR (early-abort на нечитаемом скане), worker-pool до 4 потоков Tesseract с очередью страниц, waterfall из 9 стратегий вместо одного жёсткого пути |
| `parseCache` (`collector/utils/parseCache`) | Кэш по fingerprint файла (размер + байтовые пробы) — тот же PDF второй раз не проходит re-OCR |
| Matching ShopDB (`shopDbSearch.js`) | 5 SQL-стратегий идут параллельно (`Promise.all`), не последовательно |
| Кэширование в matching | Три независимых слоя: `ShopDbQueryCache` (SQL), `agentResultCache` (результаты агента), `lineMatchCache` per-тред (follow-up «добавь позицию 5» не пересчитывает всё) |
| `SHOP_DB_SEARCH_AGENT_LLM=0` по умолчанию | Самый дорогой путь (LLM на строку) **выключен по умолчанию** — осознанное решение по стоимости, не упущение |
| Streaming + keepalive | Основной ответ стримится end-to-end; 15 с keepalive в SSE держит соединение через многоэтапный post-processing вместо обрыва |
| `skipStylePolish` для КП | Второй проход LLM не трогает числа в документе — верная защита данных |
| `OFFER_KP_MATCH_CONCURRENCY` | Осознанно низкий default (1–2), потому что «маленький pool MySQL» — не упущение производительности, а trade-off стабильность > скорость |

### 5.2 Слабые стороны / риски

1. **Matching на 100% лексика, ноль эмбеддингов.** `nameSimilarity.js` считает TF-IDF cosine + Levenshtein вручную в JS — это не семантический поиск, а продвинутое текстовое сопоставление. Синонимы, варианты транслитерации (ГОСТ/GOST), парафразы позиций вне каталога всё равно уходят в LLM-fallback (`searchAgent.pickProductsWithLlm`, per строка, последовательно). Это главный рычаг улучшения — см. 5.4.
2. **До 6+N последовательных round-trip LLM на одно сообщение**: quote-intent judge → per-line LLM fallback (если включён) → генерация → Yandex fact-check → OpenRouter/ГАРАНТ fact-check → style-polish → follow-up suggestions. Fact-check шаги (Yandex и OpenRouter/ГАРАНТ) работают над **одним и тем же** сгенерированным текстом независимо, но идут последовательно (`generation.js`, шаги 12a→12b) вместо `Promise.all` — чистая правка в коде, без новой инфраструктуры, сокращает критический путь на полный round-trip.
3. **Нет prompt/KV cache у реально используемых провайдеров.** `cache_control` (Anthropic-style prompt caching) есть только в `AiProviders/anthropic`, который прод не использует — LM Studio и OpenRouter (фактические провайдеры) ничего не имеют. Большой общий системный блок (инструкции ГАРАНТ/Yandex/Google + RAG) считается с нуля на каждый request.
4. **Один T4 16GB делится между vision и chat через полный unload/reload** (`lms unload --all` → `lms load`, `docker/lmstudio-switch.sh`). Каждое переключение OCR-vision ↔ chat — несколько–десятки секунд мёртвого времени, без перекрытия запросов.
5. **LM Studio вместо движка с continuous batching.** Нет PagedAttention/prefix caching класса vLLM или `llama-server` — при параллельных пользователях и большом общем префиксе промпта это прямая потеря пропускной способности.
6. `parseCache` только в памяти, 50 записей, restart чистит всё — для реального workflow (retry, тот же PDF клиента второй раз) кэш быстро «вымывается».
7. Хрупкий re-parsing формата каталожного блока (`autoQuoteArtifacts.parseCatalogBlock`) — уже отмечено в разделе 3, по-прежнему актуально как риск процессинга.

### 5.3 Техники ускорения — из внешнего research, привязанные к конкретному коду

**Генерация LLM**
- `Promise.all` для Yandex fact-check + OpenRouter/ГАРАНТ fact-check вместо последовательных `await`/`await` в `generation.js` — ноль новой инфры, один round-trip меньше на критическом пути.
- Prefix/KV-cache reuse: у `llama-server` (llama.cpp) есть `--prompt-cache` и slot-matching по сходству префикса промпта — в публичных бенчмарках TTFT падает с ~1,7 с до ~0,03 с при попадании в кэш. При большом, в основном постоянном system-prompt этого проекта (инструкции ГАРАНТ/Yandex/Google) это напрямую бьёт в п. 5.2.3. [Tutorial: KV cache reuse with llama-server](https://github.com/ggml-org/llama.cpp/discussions/13606)
- vLLM с `--enable-prefix-caching` даёт ~30% throughput при общем system-prompt «бесплатно» (один флаг) — рассмотреть как замену/дополнение LM Studio при многих параллельных сессиях. [vLLM Optimization for Scalable Scheduling, Batching & Concurrent Inference](https://medium.com/@abonia/vllm-optimization-for-scalable-scheduling-batching-concurrent-inference-a050f3ab1f06)

**Matching ShopDB (retrieve → rerank вместо LIKE + LLM-per-строка)**
- Классический паттерн: bi-encoder (sentence-transformers) считает эмбеддинги всего каталога **один раз**, запрос кодируется на лету, top-K по cosine за миллисекунды; cross-encoder дотягивает только top ~10 кандидатов перед возможным LLM. [Retrieve & Re-Rank — Sentence Transformers docs](https://www.sbert.net/examples/sentence_transformer/applications/retrieve_rerank/README.html)
- Каталог в MySQL и, вероятно, < 100k SKU — отдельная векторная БД не нужна. По философии проекта (все кэши in-memory) проще всего ANN in-process (напр. `hnswlib-node`/`usearch`) рядом с существующими SQL LIKE-стратегиями, а не отдельная инфраструктура. FAISS не поддерживает гибрид BM25+вектор нативно; pgvector потребовал бы Postgres, которого здесь нет. [Hybrid Search in 100 Lines: BM25 + pgvector with RRF](https://dev.to/gabrielanhaia/hybrid-search-in-100-lines-bm25-pgvector-with-rrf-merge-58cn)
- Важный вывод из research: эмбеддинги «размывают» идентификаторы/SKU/коды ГОСТ — поэтому существующие стратегии `exact_sku`/`structured`/`keywords` (лексические) **не должны** исчезнуть, а остаться «sparse»-ногой гибрида, dense (эмбеддинги) — второй ногой, слияние (RRF) в конце. Половина гибрида уже есть в коде.

**Чтение файлов / OCR**
- Tesseract.js остаётся верным выбором как CPU-bound fallback (малый footprint, работает везде) — литература это подтверждает. Ускорение имеет смысл только если у collector реально есть доступ к GPU T4 (проверить) — тогда PaddleOCR (~120 стр/мин на RTX 3090 vs ~25 стр/мин Tesseract на CPU) или Surya (лучше на layout-heavy документах) дают реальный выигрыш на тяжёлых сканах. Не внедрять вслепую — сначала замерить, как часто SmartOCRAgent вообще уходит в полный OCR (probe уже должен это ограничивать). [PaddleOCR vs Tesseract vs EasyOCR: OCR Speed and Accuracy 2026](https://www.codesota.com/ocr/paddleocr-vs-tesseract)

**GPU / quantization**
- Более агрессивная квантизация (AWQ/GPTQ 4-bit) vision-модели (`qwen3-vl-8b`, сейчас ctx 8192) могла бы уйти ниже ~5–6 GB, что при 16 GB T4 позволило бы держать **обе** модели (eyes + brain) резидентными одновременно и убрать `lms unload --all`/`lms load` на каждое переключение — вероятно, самая ценная инфраструктурная правка всего аудита, потому что снимает жёсткий синхронный stall с каждого запроса, смешивающего OCR-vision и чат. [Running Multiple LLMs Simultaneously: GPU Memory Management](https://dasroot.net/posts/2026/03/running-multiple-llms-gpu-memory-management/)

### 5.4 Помогут ли техники из описания вакансии (BERT/LoRA/PEFT, matching, quantization, multi-GPU) этому проекту?

Кратко: **да, в части matching/retrieval — почти учебное попадание в разделы 5.2.1/5.3**; остальное описание вакансии завышено относительно текущей шкалы проекта.

| Компетенция из описания | Соответствие проекту |
|---|---|
| Дообучение BERT с LoRA/PEFT | **Высокое.** Дообучение маленького bi-encoder на парах (строка заявки ↔ подобранный товар) из существующего golden set `test_files/*.expected.csv` + логов прод-matching реалистично на скромном доменном наборе — именно там LoRA лучше full fine-tuning (сообщают +1,6–4,6 пп out-of-domain vs full FT). |
| Matching-модели: генерация кандидатов, ranking, offline/online валидация | **Высокое.** Это 1:1 описание того, что сегодня вручную делает `productSearchAgent.js` (SQL LIKE + TF-IDF + LLM). Bi-encoder (кандидаты) + cross-encoder (ranking) + расширение golden set (offline валидация, уже в разделе 3.5 как «сделать») — естественное расширение роли. |
| Bi-encoder / cross-encoder | **Высокое.** Прямо закрывает 5.2.1 — см. 5.3. |
| Оптимизация инференса: quantization, distillation | **Среднее/высокое.** Прямо бьёт в проблему общего T4 (5.2.4) — quantization vision-модели, возможно distillation локальной agent-модели под меньший footprint. |
| Multi-GPU / distributed training | **Низкое здесь и сейчас.** Проект на одном T4 только для инференса; дообучение маленького bi-encoder LoRA помещается на одном GPU (даже том же T4, вне пика). Важно только при реальном тренинге больших моделей с нуля. |
| Uplift-modeling, highload ML-системы | **Низкое.** Это чатбот-КП для одного магазина, не система рекомендаций/прайсинга с treatment/control на большой шкале. |

**Вывод для найма:** если вакансия финансирует работу над этим репо, взвешивайте критерии в сторону matching/retrieval + PEFT (это реальный, выявленный техдолг), а multi-GPU/uplift/highload считайте «nice to have», не «must have» — иначе риск нанять профиль, не совпадающий с фактической шкалой задачи.

### 5.6 Внедрение: лёгкий embedding-reranking в matching (2026-07-21)

Добавлен как **дополнительный, опциональный слой поверх** существующего TF-IDF/Levenshtein/Jaro-Winkler — структура `nameSimilarity.js`/`shopDbSearch.js` (SQL LIKE, кластеризация `productsAreSimilar`/`pickCheaperAmongSimilar`) не тронута.

- **Новый файл:** `server/utils/offerKp/embeddingSimilarity.js` — модель `MintplexLabs/multilingual-e5-small` через `@xenova/transformers` (та же библиотека и механизм загрузки/fallback, что у существующего `NativeEmbedder` в `server/utils/EmbeddingEngines/native`, переиспользован через подкласс с переопределённым `getEmbeddingModel()` — ноль правок в существующем RAG-embedder). CPU-only, не трогает GPU/LM Studio/T4.
- **Точка встройки:** `nameSimilarity.js#searchByNameSimilarity` — новая функция `applyEmbeddingBoost` переранжирует **уже отфильтрованных TF-IDF** кандидатов (не сырой пул SQL LIKE, остаётся «лёгким»), смешивая `_nameSimilarity = max(base, base·(1-w) + cosine·w)`, по умолчанию `w=0.3`.
- **Кэш:** отдельная in-memory карта эмбеддингов товаров (TTL 24ч, 4000 записей) в том же стиле, что `ShopDbQueryCache`/`lineMatchCache` — названия товаров меняются редко, повторные запросы не re-embed'ят весь каталог.
- **Fail-safe:** любая ошибка (нет сети при первой загрузке модели ~487MB и т.п.) отключает embedding-boost на остаток процесса, pipeline **молча возвращается к чистому TF-IDF** — поведение как до правки. Kill-switch: `SHOP_DB_EMBEDDING_SIMILARITY=0`.
- **Env vars:** `SHOP_DB_EMBEDDING_SIMILARITY`, `SHOP_DB_EMBEDDING_MODEL`, `SHOP_DB_EMBEDDING_WEIGHT`, `SHOP_DB_EMBEDDING_MAX_CANDIDATES`, `SHOP_DB_EMBEDDING_CACHE_TTL_MS`, `SHOP_DB_EMBEDDING_CACHE_MAX_ENTRIES` — см. `server/.env.example` (секция ShopDB).
- **Тесты:** `server/__tests__/utils/offerKp/embeddingSimilarity.test.js` (kill-switch, пустые входы — без сети). Полный `server/__tests__/utils/offerKp/*` (367 тестов, включая `goldenSet.test.js`) проходит без изменений.
- **Осознанный scope:** reranking работает только по кандидатам, которых SQL LIKE вообще нашёл (есть общий токен) — это **не** полный семантический retrieve кандидатов (эмбеддинг как первая нога «retrieve»), а улучшение порядка/качества среди уже найденных. Полный retrieve-by-embedding (ANN над всем каталогом) — большая правка, всё ещё открыта в 5.3/5.5.

## 6. Цикл обучения без fine-tuning + аудит «Чтение документа...» (2026-07-21, дополнение 2)

### 6.1 Golden set как источник правок matching (без тренировки моделей)

Контекст: `test_files/*.expected.csv` до сих пор валидировал **только экстракцию** (`source_name/unit/quantity`, см. `test_files/README.md`) — ничего из добавленных примеров не влияло на качество сопоставления с каталогом. Добавлены два механизма, которые реально «учатся» на примерах без тренировки любой модели:

1. **Таблица коррекций (override)** — `server/utils/offerKp/goldenCorrections.js`. Расширяет существующую схему CSV опциональными колонками `matched_sku,matched_name,match_type` (exact/analog/none) — файлы без этих колонок (чисто экстракционные, как сейчас) игнорируются, ничего не ломается. Встроено в `matchInquiryLines.js#matchInquiryLine`: точная (нормализованная) повторённая строка запроса → авторитетный ответ оператора, проверяется ДО живого поиска. Цена/название всегда подтягиваются живьём из ShopDB по SKU (`searchByExactSku`) — golden set никогда не источник цены, только указывает **какой** товар. Если SKU из golden set уже нет в каталоге — pipeline падает обратно в обычный поиск (не молчит как «нет совпадения»).
2. **Few-shot retrieval для LLM fallback** — `server/utils/offerKp/goldenFewShot.js`. Переиспользует эмбеддинг из 5.6: эмбеддит новую строку запроса, достаёт k (по умолчанию 3, порог сходства 0.55) наиболее похожих ПОЛОЖИТЕЛЬНЫХ примеров из golden set и вставляет их в промпт `searchAgent.pickProductsWithLlm` как «так мы раньше сопоставляли похожие случаи». Чем больше примеров в golden set, тем точнее подсказки — без тренировки чего бы то ни было.
3. Автокалибровка порогов (`SHOP_DB_EMBEDDING_WEIGHT`, `SHOP_DB_NAME_SIMILARITY_MIN` и т.д.) offline-скриптом под golden set — **не сделана**, остаётся следующим шагом, когда golden set вырастет.

**Env vars:** `SHOP_DB_GOLDEN_CORRECTIONS` (kill-switch), `SHOP_DB_FEW_SHOT_EXAMPLES`, `SHOP_DB_FEW_SHOT_MIN_SIMILARITY`. **Тесты:** `goldenCorrections.test.js`, `goldenFewShot.test.js` — полный `server/__tests__` (526 тестов) зелёный.

**Чтобы это заработало**, записи в `test_files/*.expected.csv` должны иметь заполненные `matched_sku`/`match_type` — одни `nr,source_name,unit,quantity` (текущее состояние файлов) ничему не учат.

### 6.2 Root cause: «Чтение документа...» висит очень долго

Во время тестов нашлись **незакоммиченные, но полные** изменения в рабочем дереве (не мои — WIP из более ранней сессии), которые уже бьют ровно в эту проблему. Комментарий в коде говорит прямо:

> «Swapping Qwen Thinking ↔ gpt-oss costs 90+ seconds and can unload a model from an active request.»

То есть: прежний «быстрый sequential switch eyes/brain» (последний коммит до той сессии) всё ещё стоит **90+ секунд** на каждое переключение vision-модели (OCR) ↔ agent на одном T4 16GB — и это главная причина, почему «Чтение документа...» висит. WIP меняет архитектуру на **одну резидентную модель** (`qwen/qwen3-vl-8b` для обеих ролей — OCR и chat/agent), убирая переключение совсем: `offerKpModelPipeline.js`, `offerKp.llm.defaults.js`, `offerKp.models.js`, `normalizeWorkspaceLlms.js` (новый `OFFER_KP_SINGLE_MODEL`), `docker/lmstudio-switch.sh`/`lmstudio-load-brain.sh`, `scripts/deploy-lainey-sync.sh` (дописывает профиль T4 в `.env` прода + ставит `tesseract`/`poppler-utils` при отсутствии), плюс новые этапы прогресса в UI (`vision-ocr`, `pipeline-agent-load`).

**Дополнительно найден и исправлен баг в том же WIP** (всплыл как failing test `inquiryTextQuality.test.js` → «accepts a complete structured table»): `assessInquiryTableIntegrity` в `offerKpDocumentIngest.js` проверял наличие единицы (`кг/шт/м/уп`) в `line.raw` — но `parseInquiryText` для структурных таблиц **уже выносит единицу** в отдельное поле `line.unit`, поэтому условие всегда fail'ило даже на идеально чистой таблице. Эффект: `needsReocr=true` на хорошем тексте → **лишний** вызов дорогого `enrichDocumentsWithOfferKpOcr` (vision-модель, ещё секунды/десятки секунд) на документах, которым это не нужно — вторая, независимая причина того же симптома.

Правка (`offerKpDocumentIngest.js`): вместо хрупкого regex на `raw` детектится **over-segmentation** — если `parseInquiryText` вернул заметно больше логических строк, чем число совпадений ключевых слов в сыром тексте (`parsed.length > candidateRows * 1.15`), значит один товар разъехался на несколько строк (типичный эффект плохого OCR/переносов) и **ни одной** строке не доверяем — независимо от того, выглядит ли она поодиночке полной. Проверено на обоих существующих golden-тестах целостности таблицы (чистая → `usableRows=3/3`, сломанная → `usableRows=0/3`, оба совпадают с ожиданиями тестов).

**Прод-логи (Selectel Lainey) в этой сессии не проверялись** — нет SSH в этой среде; root cause установлен статически по комментариям в WIP-коде + по failing тесту, не по live-логам. Если после деплоя проблема останется, `offerkp logs` / `journalctl -u offer-kp` на Lainey покажет, отключил ли `OFFER_KP_SINGLE_MODEL` переключение моделей на проде.

## 7. MD-файлы как «skill»/harness для дообучения без дообучения (2026-07-21)

Вопрос: помогает ли здесь техника а-ля Claude Agent Skills (markdown-файлы с progressive disclosure) — ещё один способ «учить» систему без тренировки весов?

### 7.1 Как это работает у Anthropic (для сравнения)

Skill — это папка с `SKILL.md`, который начинается с YAML frontmatter (`name`, `description`). Работает в три слоя ([Anthropic: Equipping agents with Agent Skills](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills), [Agent Skills docs](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)):

1. **Discovery** — при старте в системный промпт грузятся только `name`+`description` каждого скила (~80 токенов на скил в среднем) — почти бесплатно, даже при 50+ установленных скилах.
2. **Основной контент** — полное тело `SKILL.md` (до ~5000 токенов) подгружается, только когда описание совпало с задачей.
3. **Вложенные ресурсы** (`REFERENCE.md`, скрипты) — грузятся ещё позже, только если сама skill решит, что они нужны.

Ключевая идея — **progressive disclosure**: дешёвый индекс всегда виден, дорогой контент подгружается только по релевантности, а не пихается в каждый промпт.

### 7.2 Это НЕ «обучение» — это третий, отдельный инструмент рядом с уже сделанным

Важно не путать: то, что сделано в §6.1 (override-таблица + few-shot), — это **память о конкретных примерах** (query → правильный товар). Skill-подобный markdown — это **явные процедурные правила** (политики), которые пишет человек, а не собирает система из логов. Это ортогональные вещи:

| Механизм | Что содержит | Кто наполняет | Когда полезен |
|---|---|---|---|
| Override table (§6.1.1) | Точные пары запрос→SKU | Растёт из golden set / логов | Повторяющийся ровно тот же запрос |
| Few-shot retrieval (§6.1.2) | Похожие прошлые примеры | То же | Похожий, но не идентичный запрос |
| **MD-skill/knowledge (новое)** | Явные правила/политики домена | Человек-эксперт пишет прозой | Систематическая неопределённость, которую примерами не закрыть |

### 7.3 Конкретный кейс, где это реально нужно — только что найден в §6.4

20 строк `Slozhnost_vysokaya_1` остались без матчинга не из-за нехватки примеров, а потому что **в тексте заявки в принципе нет данных** (класс прочности, покрытие) — никакой RAG и никакой fine-tuning это не восстановит, потому что информации физически нет во входе. Единственный способ — явное **правило по умолчанию**, которое сегодня существует только у человека в голове: например «если класс прочности/покрытие не указаны в заявке на ГОСТ 7805-70 — предлагать оцинкованный 8.8 как наиболее ходовой вариант, но матчить как `similar`, не `exact`, с пометкой для подтверждения». Это ровно то, что должно жить в markdown-skill, а не в примерах.

### 7.4 Как это встроить технически (без переписывания промпт-архитектуры)

В проекте уже есть подходящая структура — `server/utils/offerKp/prompts.js`, `catalogPrompt.js`, условные блоки инструкций по env/контексту (ГАРАНТ/Яндекс/Google уже собираются так же). Прямое расширение:

1. `server/utils/offerKp/knowledge/*.md` — по файлу на тему (`din-gost-equivalents.md`, `strength-coating-defaults.md`, `analog-substitution-rules.md`), с коротким YAML-заголовком `description` для discovery-слоя.
2. Триггер загрузки — **уже существующие сигналы**, не новый классификатор: `parsed.dinNumbers`, `detectAnalogIntent()`, `hasHardwareSignals()` из `hardwareQuery.js`/`productSearchAgent.js` уже определяют, есть ли в запросе DIN/ГОСТ/аналоговый интент — этого достаточно, чтобы решить, подгружать ли конкретный knowledge-файл в промпт `searchAgent.pickProductsWithLlm`/`quoteIntentJudge`.
3. Полный файл (не только заголовок) подмешивается в system-промпт **только когда сигнал сработал** — это и есть progressive disclosure, без него каждый knowledge-файл рос бы в промпте каждого запроса и раздувал контекст (та же проблема, что уже отмечена в §5.2.3 — большой общий системный промпт без prefix-кэша).
4. Формат правок — **обычный markdown, который редактирует человек без доступа к коду** (catalog-менеджер, не разработчик) — важное отличие от override-таблицы, которая требует знать SKU и трогать CSV с точным форматом.

### 7.5 Риски

- Правила в прозе легко становятся противоречивыми при росте файла — нужен явный владелец контента и ревью, иначе это тот же дрейф, что у любого промпт-инструктажа без версионирования.
- Это НЕ замена структурным данным: если правило можно выразить как код (`analogRules.js` уже делает часть этого детерминированно) — код надёжнее прозы для LLM. MD-skill имеет смысл именно там, где решение — это осознанный бизнес-компромисс («что предлагать по умолчанию, когда данных не хватает»), а не техническая логика сопоставления.

**Вывод: да, помогает — но не как замена fine-tuning/embedding, а как третий слой поверх уже сделанного (§6.1), закрывающий конкретно найденный пробел (систематическая неполнота входных данных, а не нехватка примеров).**

**Обновление (2026-07-21, дополнение 4): реализовано.** См. §8.2 — `knowledgeBase.js` + 2 файла (`din-gost-equivalents.md`, `strength-coating-codes.md`), подключено в `searchAgent.pickProductsWithLlm`.

## 8. «Делай всё, чтобы улучшить поиск» (2026-07-21, дополнение 4)

Задача без ограничения объёма — сделан набор изменений от самых дешёвых (расширение уже сделанного) до экспериментальных (новая модель), с измерением на живой ShopDB на каждом шаге. Инструмент измерения: `scripts/measure-shopdb-search-quality.cjs` — прогоняет `runProductSearchAgent` (без golden-override) по всем верифицированным строкам golden set (§6.4) и считает accuracy@1/@5.

### 8.1 Расширение embedding-реранкинга (nameSimilarity.js, embeddingSimilarity.js)

Из §5.6: реранкинг раньше применялся только к кандидатам, УЖЕ прошедшим порог TF-IDF (`DEFAULT_MIN_COSINE=0.32`) — парафраз с низким лексическим пересечением не имел шанса, даже если embedding его узнавал.

- `searchByNameSimilarity` теперь скорит **весь** пул SQL LIKE (`minScore=0`), фильтрует **после** блендинга с embedding, не до.
- Новый порог `SHOP_DB_EMBEDDING_STANDALONE_MIN` (по умолчанию 0.62): кандидат проходит, если лексический score ≥ 0.32 **ИЛИ** embedding-score ≥ 0.62 — это OR, не AND, так что сильный семантический сигнал сам по себе спасает кандидата с нулевым лексическим пересечением.
- `SHOP_DB_EMBEDDING_MAX_CANDIDATES` поднят с 40 до 80 (весь дефолтный пул), иначе расширение фильтра было бы бессмысленным — кандидаты за пределами топ-40 по TF-IDF просто не доехали бы до эмбеддера.

### 8.2 MD-knowledge harness — реализован (§7 → готово)

`server/utils/offerKp/knowledgeBase.js` + `server/utils/offerKp/knowledge/{din-gost-equivalents,strength-coating-codes}.md`. Подключено в `searchAgent.pickProductsWithLlm`: сигналы `parsed.standardNumbers`/`detectAnalogIntent()` (уже существующие, ничего нового не классифицируем) решают, подмешивать ли файл в промпт. Тесты: `knowledgeBase.test.js`.

### 8.3 Два реальных бага в детерминированном поиске (найдены при живом измерении, не гипотеза)

Первый прогон measure-скрипта показал **43% accuracy@1** — подозрительно низко для структурного SQL-поиска по полностью специфицированным DIN-запросам (§6.4). Раскопки на живой БД нашли два независимых, ранее незамеченных бага:

**Баг 1 — LIKE-паттерны резьбы никогда не совпадали.** `searchByStructuredQuery` (`shopDbSearch.js`) и `extractFuzzyTerms` (`searchAgent.js`) искали резьбу как `%M 8x14%`/`%M8x14%` и т.п. Каталог форматирует резьбу с выравниванием по ширине и переменным числом пробелов: `"M  8x 14"`, `"M 16x 70"`, `"M 10x100"` (без пробела при 3-значной длине). Проверено на живой БД: **все 4 старых паттерна — 0 совпадений**, ни одного, при любом размере. Правка: `REGEXP` с `[[:space:]]*` вместо фиксированных LIKE — проверено, находит совпадения для всех протестированных размеров.

**Баг 2 — клиент написал «винт», ГОСТ говорит «болт».** DIN 933/931 (ГОСТ 7805-70/7798-70) — это болты, но разговорный текст заявки часто называет их «винт» (бытовое смешение терминов). `searchByStructuredQuery` фильтрует по `productTypes` через **AND**, так что при слове «винт» в запросе товары с именем «Болт DIN 933…» отбрасывались категорически, несмотря на точное совпадение DIN+резьбы. Правка: `STANDARD_IMPLIES_TYPE` в `hardwareQuery.js` — известные стандарты (933/931/7805/7798→болт, 912/11738→винт, 934/5915→гайка) добавляют канонический тип **в дополнение** к тому, что написал клиент (объединение множеств, не замена).

**Эффект (измерено, n=7, см. предупреждение о размере выборки ниже):**

| Этап | accuracy@1 | accuracy@5 | не найдено |
|---|---|---|---|
| Baseline (до правок) | 43% (3/7) | 43% (3/7) | 4/7 |
| + Баг 1 (REGEXP резьба) | 57% (4/7) | 71% (5/7) | 2/7 |
| + Баг 2 (винт→болт) | 57% (4/7) | **86% (6/7)** | 0/7 |

### 8.4 Эксперимент: cross-encoder reranker (по запросу — «легкий embedding, не вешать T4»)

Дискуссия про Qwen3-Reranker-0.6B: главная мысль верна — bi-encoder (embedding, §8.1) кодирует запрос и товар НЕЗАВИСИМО, поэтому плохо различает «похожий, но неверный» вариант (M16x70 vs M16x40 — оба близки в векторном пространстве). Cross-encoder читает пару вместе, лучше видит несовпадение конкретных чисел.

**Почему не Qwen3-Reranker-0.6B буквально:** его ONNX-конвертация ориентирована на `@huggingface/transformers` v4/`text-ranking` pipeline; в проекте стоит `@xenova/transformers` v2.14 (та же либа, что уже используется для embedding-реранкинга §5.6/8.1) — апгрейд major-версии общей embedding-инфраструктуры ради одной экспериментальной фичи непропорционален риску. Вместо этого — `Xenova/bge-reranker-base` (XLM-RoBERTa, официальная конвертация Xenova, тот же namespace что и уже работающая `MintplexLabs/multilingual-e5-small`, ~278MB, CPU-only, не трогает GPU/LM Studio/T4) через низкоуровневый API (`AutoTokenizer`+`AutoModelForSequenceClassification`+`text_pair`), не через pipeline (которого может не быть в v2.14).

Реализовано в `server/utils/offerKp/crossEncoderRerank.js`, подключено в `productSearchAgent.js` (переранжирует уже найденный топ, не весь каталог). **Выключено по умолчанию** (`SHOP_DB_RERANKER_ENABLED=0`) — это прямо повторяет рекомендацию из обсуждения: сравнить, а не сразу включать.

Живая проверка в этой сессии: модель качается и считает логиты (сеть в этой песочнице оказалась доступна, ранее казалось иначе из-за отдельного сбоя нативного модуля `sharp`, не сети). На синтетическом примере (Болт M16x70 vs Болт M16x40 vs Гайка M16) reranker уверенно топит категориально неверный кандидат (гайка → 0.002) и держит оба болта высоко (0.999 / 0.979) — то есть хорошо ловит грубые ошибки типа изделия, но НЕ является заменой точной валидации размеров (структурный SQL-фильтр §8.3 остаётся основным источником точности по числам).

**Результат на golden set (n=7, `SHOP_DB_RERANKER_ENABLED=1`):**

| Этап | accuracy@1 | accuracy@5 |
|---|---|---|
| После фиксов §8.3, без reranker | 57% (4/7) | 86% (6/7) |
| + cross-encoder reranker | 57% (4/7) | **100% (7/7)** |

Оба DIN933-примера (M16x70, M16x40) сдвинулись вверх (были #6/#4, стали #3/#4), но не дотянули до #1 — reranker улучшает recall в топ-5, не решает точное ранжирование единственного правильного варианта для этих двух случаев. Долетел до 100% на @5 только последний оставшийся «не найдено» случай.

**Важная оговорка про стоимость:** прогон 7 запросов с reranker'ом занял **~6 минут CPU-времени** (без reranker — секунды). Cross-encoder не кэшируется как embedding (каждая пара запрос+кандидат — отдельный проход модели, до 15 кандидатов на запрос по умолчанию) — это на порядки дороже per-query, чем bi-encoder из §8.1. С учётом этого `SHOP_DB_RERANKER_ENABLED=0` по умолчанию — не просто осторожность «сначала сравнить», а обоснованное решение: в текущем виде включение по умолчанию на каждый запрос ощутимо увеличит латентность matching. Если включать — то выборочно (напр. только когда лексический топ-1 и топ-2 близки по score), не на каждый запрос; эта логика не реализована.

### 8.5 Важная оговорка о размере выборки

Все цифры выше — n=7 (единственные верифицированные строки golden set, §6.4). Это снимок, не статистически значимый результат — с 7 примерами один правильный/неправильный кейс двигает accuracy на 14 п.п. Числа показывают НАПРАВЛЕНИЕ (баги реальны, фикс не сделал хуже), не точную величину эффекта в проде. Приоритет №1 для следующей итерации — расширить golden set matching-примерами (§6.1/6.4), иначе дальнейшая калибровка на этих же 7 строках рискует переобучиться на шум.

### 8.6 Что осталось нетронутым (сознательно)

Из обсуждения: hard negatives из логов оператора, отдельный matching-бенчмарк (`positive_sku,allowed_analog_skus,forbidden_skus`), instruction-aware embedding для Qwen3-Embedding, полный ANN над каталогом, SGLang — не реализовано. Причина одна и та же для всех: либо нужны данные, которых пока нет (логи корректировок оператора не собираются — см. уже отмеченное в test_files/README.md), либо это оптимизация для масштаба, которого у проекта пока нет (сотни параллельных поисков). Согласен с порядком приоритетов из обсуждения: сначала данные и структурная точность (§8.3 — сделано), потом реранкер как эксперимент (§8.4 — сделано, не включено), LoRA fine-tuning — только после накопления сотен реальных корректировок.

Прямое следствие этого пункта — §9 ниже: чтобы «логи корректировок оператора» вообще появились, нужен непрерывный сбор данных, а не разовые замеры.

## 9. Непрерывный сбор retrieval-метрик + `offerkp metrics` (2026-07-21, дополнение 5)

Из §8.6: matching-бенчмарк и hard negatives упираются в то, что операционных данных попросту нет — `scripts/measure-shopdb-search-quality.cjs` (§8) меряет только 7 golden-примеров разово, вручную. Нужен триггер, который сам, непрерывно, без участия человека, копит данные о качестве поиска в проде.

### 9.1 Механизм (сервер)

`server/utils/offerKp/searchMetrics.js` — `recordSearchMetric(event)`, fire-and-forget дозапись JSON-строки в `storage/metrics/shopdb-search.jsonl` (`STORAGE_DIR`-относительно, как остальной storage). Никакой новой инфраструктуры: без сети, без схемы БД, без модели — просто `fs.appendFile`. Ротация: один backup-файл при превышении `SHOP_DB_METRICS_MAX_BYTES` (по умолчанию 20MB).

**Триггер** — вызов `recordSearchMetric` вшит прямо в `matchInquiryLine` (`matchInquiryLines.js`), центральную функцию, через которую проходит **каждая** строка **каждой** заявки в проде:

- Успешный путь (перед `return matchedLine`): `matchType`, `source` (стратегия, давшая лучший результат — `golden_override`/`exact_sku`/`structured`/`name_cosine`/`llm_rank`/…), полный список сработавших `strategies`, `hasPrice`, число кандидатов, длина запроса, `threadId`.
- Путь ошибки (`buildLineMatchErrorFallback`): отдельное событие `matchType: "error"` — чтобы видеть не только «не нашли», но и «упало технически» (обрыв MySQL и т.п., см. риск §3.5 в основном аудите).

Ничего не блокирует matching: запись асинхронная, обёрнута в try/catch, ошибка записи только логируется и не пробрасывается наверх — тот же принцип graceful degradation, что и у embedding/golden-override.

**Живая проверка в этой сессии:** вызвал `matchInquiryLine` напрямую с реальными строками, подтвердил появление корректных JSON-записей (`{"matchType":"exact","source":"search_index","strategies":[...],...}`) в файле.

### 9.2 Агрегация

`scripts/report-shopdb-metrics.cjs [--hours N] [--tail N]` — читает JSONL (+ backup), фильтрует по окну времени, печатает разбивку по `matchType`, по стратегиям (сколько раз каждая реально сработала), % с ценой, % ошибок, среднее число кандидатов, % через golden override, % потребовавших `llm_rank`. Живой прогон на 2 синтетических событиях — отчёт корректен (см. вывод команды в этой сессии).

### 9.3 CLI (`cli/`, Go)

Новый таб `offerkp metrics` (клавиша `6`), `offerkp metrics --plain [--hours N]` — по образцу существующих Build/CI-CD табов (`cli/metrics.go`, правки `main.go`/`tui.go`/`config.go`). Механика: SSH на Lainey (`sshRun`, как и остальные табы), `cd $OFFERKP_REMOTE_APP && node scripts/report-shopdb-metrics.cjs --hours N` (новая env `OFFERKP_REMOTE_APP`, по умолчанию `/opt/offer-kp/app` — реальный путь кода на сервере, см. `deploy-lainey-sync.sh`). Живая проверка: `offerkp metrics --plain` реально подключился по SSH к 87.228.90.43 и корректно сообщил, что скрипт ещё не задеплоен (ожидаемо — задеплоится с этим коммитом). Опрос в TUI-режиме реже, чем логи/CI-CD (раз в 20с), потому что каждый опрос — это SSH + отдельный процесс `node` на сервере, не бесплатная операция.

### 9.4 Область и ограничения

- Метрики **описывают** качество (что нашли/не нашли), но не автоматически **исправляют** его — это фундамент для будущего matching-бенчмарка и hard-negative mining из §8.6, не замена им.
- В логе нет цен, нет текста запроса целиком — только `queryLen` (число), `matchType`, стратегии, счётчики. Осознанно: это операционная телеметрия, не то же самое, что PII/коммерческие данные заявки.
- `SHOP_DB_METRICS_ENABLED=0` выключает полностью — сбор данных, в отличие от embedding/reranker, ничего не стоит по латентности (локальная асинхронная запись в файл), поэтому включён по умолчанию, в отличие от reranker'а (§8.4).
- Не реализовано: экспорт из JSONL в структурированный matching-бенчмарк/датасет для LoRA — следующий логичный шаг, когда накопится достаточно данных.

## 10. Прод-баг «выдуманный каталог на мета-вопрос» + guardrails/grounding аудит (2026-07-21, дополнение 6)

### 10.1 Репорт

Пользователь на **первом сообщении** без вложенного PDF написал мета-вопрос: «Почему в КП нет цен? Подставь каталог purolat.com и переделай DOCX с позициями и SKU». Ответ ассистента — три полностью выдуманных товара (`Штанга DIN 975 M36×2000...`, фейковые SKU вида `975M362000Z`, цены, ссылки на `purolat.com/product/...`) в формате, имитирующем реальный блок `[Каталог · purolat.com]`, плюс ложное заявление «Документы DOCX и PDF... уже готовы».

### 10.2 Разбор по слоям (что сработало правильно, что нет)

Проверено трассировкой фактического кода, не догадкой:

- **intentRouter.js сработал верно.** `routeOfferKpMessage(...)` для этого текста даёт `primaryIntent=edit_quote`, `intents=[edit_quote, product_search]` (за счёт паттерна «подставь ... каталог»), политика `allowShopDbSearch=allowQuoteMutation=allowCatalogPriceUse=allowExport=true` — это именно то, что должно быть для «переделай КП с реальными данными». Совпадает с уже существующей строкой golden-фикстуры `intentRouting.jsonl:184` и тестом `quoteIntentJudge.test.js:70-76`. **Гипотеза «сломан intent-роутинг» не подтвердилась** — реальный поиск ShopDB для этого сообщения действительно разрешён.
- **ShopDB-поиск легитимно не нашёл ничего**, потому что в сообщении нет ни одного технического параметра (DIN/ГОСТ/размер/тип) — искать физически нечего. Это ожидаемое поведение, не баг.
- **Anti-hallucination инфраструктура (уже существующая, и она серьёзная) не сработала — но не потому что она сломана, а потому что её область покрытия не включала этот случай.** В проекте уже были: детерминированная проверка цен по markdown-таблице (`quoteDbPriceGate.js`), compliance-чекер для `create-docx-file`/`create-pdf-file` (`quoteComplianceChecker.js`), CRAG-подобная оценка полноты каталога с порогами абстинирования (`harnessEvidence.js: gradeCatalogEvidence/shouldAbstainFromEvidence`), обязательная верификация источника перед генерацией файла (`offerKpSourceVerificationBlock.js`), и даже отдельный LLM-судья на approve тулов `create-docx/pdf` (`quoteIntentJudge.js`). **Все они гейтят вызов инструмента** (`beforeToolApproval` для `create-docx-file`/`create-pdf-file`). Финальный текст **в чате** проходит только через `AgentHarness.sanitizeOutgoingChat(content)` (`aibitat/index.js:850-852`) — и там была ранняя заглушка `if (!text.includes("|")) return content;`. Формат, которому реально учит `prompts.js` («**Товар:** … / **Цена:** … / **Артикул / SKU:** …») и который использовала модель в баге — bullet-список, без символа `|`. Значит **вся построчная проверка цен полностью пропускалась** для любого ответа без markdown-таблицы — ровно то, что модель и написала.

### 10.3 Исправления (все покрыты тестами, полный набор `npx jest` — 753/753 green)

1. **`server/utils/agentHarness/AgentHarness.js` — `sanitizeOutgoingChat`.** Убрана ранняя заглушка по `|`. Добавлена независимая от формата проверка: любое вхождение `Цена: NNN` в исходящем тексте (bullet-список, `[Каталог · …]`-блок или таблица) собирается регэкспом и сверяется с реально подставленными сервером ценами (`collectAllowedPricesFromCatalog` + `collectAllowedPricesFromDraft` — уже существующие, переиспользованы как есть). Если хоть одна заявленная цена не находит соответствия — весь ответ заменяется на `ABSTAIN_MESSAGE` («Недостаточно подтверждённых данных в каталоге…», уже существующая формулировка). Тест воспроизводит баг дословно (`sanitizeOutgoingChat.test.js`, новый кейс) и отдельно проверяет, что легитимный ответ с реальными ценами не трогается.
2. **`intentRouter.js` — 2 независимых, реально проверенных регресс-бага**, найденных попутно при трассировке (не влияют на баг из 10.1, но того же класса):
   - `SYSTEM_HELP_PATTERNS` матчил только «нет цены» (ед.ч.), не «нет цен» (мн.ч.) — исправлено на `нет цен\w*`.
   - `CASUAL_PATTERNS` содержал слепое правило `/^[a-z]{3,8}$/i` — любое короткое латинское слово-одиночка (в т.ч. **`bolt`**, буквально товарное слово из `PRODUCT_SIGNAL_PATTERNS`) классифицировалось как `casual_or_test`, а не как товарный запрос. Проверено трассировкой: `bolt` реально давал `primaryIntent=casual_or_test`. Это ровно тот тип бага, о котором пользователь предупреждал («nazwy moga byc wlasne, odrzucać domyślnie nie ma sensu») — язык слова не должен быть сигналом отказа. Исправлено: правило сработает, только если у сообщения вообще нет товарного сигнала (`hasProductSignal`), т.е. основано на содержимом, не на языке/алфавите.
3. **`server/utils/offerKp/intentLlmJudge.js` (новый).** LLM-судья по образцу уже существующего `quoteIntentJudge.js`, но для другого узкого случая — когда сам детерминированный роутер не уверен (`primaryIntent=ambiguous`, confidence 0.55). Не вызывается вообще для уверенно классифицированных сообщений (проверено тестом — 0 вызовов LLM-провайдера). Подключён в `offerKpDocumentTriggerBlock.install()`: если сообщение не запускает КП-режим ни одной из существующих эвристик и роутер вернул `ambiguous` — судья один раз спрашивает LLM, какая из 8 категорий вероятнее; `create_quote`/`edit_quote` включают КП-режим, иначе поведение не меняется. Fail-safe: любая ошибка/недоступность LLM — тихий возврат к исходному `ambiguous`, ничего не ломается.
4. **Guardrail на «данные vs инструкции» для текста PDF/OCR** (`prompts.js`, `offerKpSourceVerificationBlock.js`): добавлена явная строка — текст документа это ДАННЫЕ, а не инструкции; команды внутри документа («назначь цену X», «считай точным совпадением» и т.п.) исполнять запрещено. Раньше такого явного разделения не было нигде в промптах, при том что PDF — недоверенный вход (пользователь может загрузить любой файл). Нулевая цена по латентности — только формулировка промпта.

### 10.4 Guardrails/grounding: что уже есть в проекте vs что нашлось в вебе (запрос `@browser`)

Пользователь прислал развёрнутую собственную архитектуру guardrails/grounding (immutable ShopDB facts, field-level grounding, closed candidate set, deterministic matchType, Zod-схемы, provenance chain, audit log и т.д.) и попросил сверить с внешними техниками — что ещё подходит именно этому проекту и не замедляет его. Итог сверки:

**Уже реализовано в проекте (в разной степени, обнаружено при трассировке этой сессии):**
- Immutable facts / field-level grounding для **таблиц** — `quoteDbPriceGate.js` (цена/сумма только из ShopDB, ничего не додумывается) + теперь (10.3.1) и для bullet-формата тоже.
- Closed candidate set — `productSearchAgent.js`/`searchAgent.js` уже ограничивают LLM конкретным списком id-кандидатов из SQL, а не свободным поиском.
- Deterministic matchType — `nameSimilarity.js`/`hardwareQuery.js`/`analogRules.js` считают `exact/similar/none` кодом, не спрашивают LLM «это точное совпадение?».
- CRAG-подобный grounding-грейдинг с порогом абстинирования — `harnessEvidence.js` (`gradeCatalogEvidence`/`shouldAbstainFromEvidence`, уже с 2026-07-21 дополнения 4).
- Retrieve→constrain→verify→abstain→scope layered guidelines — `config/offerKp.harnessAntiHallucination.js`.
- Audit log — частично: `searchMetrics.js` (§9) логирует matchType/стратегию/кандидатов, но не полный provenance-объект (нет `retrievalScores`/`candidateIds` в записи).
- Tool guardrails (least privilege) — частично: `QUOTE_FORBIDDEN_RETRIEVAL_TOOLS` убирает `rag-memory`/`web-scraping`/`web-browsing` из функций агента в КП-режиме (`offerKpDocumentTriggerBlock.js`), но выделенных узких тулов вида `searchShopDbProducts({standard, diameter, length})` вместо одного общего поиска — нет; это архитектурно более крупная работа.

**Новое, что нашлось в вебе и реально подходит (лёгкое, не создаёт задержек):**
- **Grammar-constrained decoding / structured output через JSON Schema** (XGrammar/llguidance, интегрированы в vLLM/SGLang/TensorRT-LLM, ~40-50 микросекунд/токен — не источник хаоса) — устраняет целый класс галлюцинаций (выдуманные ключи JSON, невалидные значения enum) **на уровне семплера**, а не пост-фактум валидацией.
  **Проверено вживую в этой сессии (SSH на Lainey, прод-сервер LM Studio, 87.228.90.43):** `curl .../v1/chat/completions` с `response_format:{type:"json_schema", json_schema:{strict:true, schema:{...}}}` на реально загруженной проде-модели `qwen/qwen3-vl-8b` вернул **валидный, точно соответствующий схеме JSON** (`{"sku":"...","price":41.25}`, без лишних полей) — LM Studio-сервер constrained-decoding **поддерживает и он работает прямо сейчас**, вопрос закрыт, это не гипотетическая возможность.
  Но: код-провайдер в проекте (`utils/AiProviders/lmStudio/index.js:222` `getChatCompletion(messages, { temperature })`) сегодня **не прокидывает** `response_format` дальше — деструктурирует только `temperature`, всё остальное из вызывающего кода теряется. Т.е. барьер не в сервере (он готов), а в узком месте адаптера. **Не внедрено в этой сессии** — нужно (1) научить `getChatCompletion`/`streamGetChatCompletion` принимать и форвардить `response_format` в `this.lmstudio.chat.completions.create(...)`, (2) завести JSON Schema для мест, где LLM отдаёт структурированное решение (`productSearchAgent.js` ранжирование кандидатов, `intentLlmJudge.js`/`quoteIntentJudge.js` категория/да-нет), (3) прогнать против **обеих** реально используемых прод-моделей (`qwen/qwen3-vl-8b` и `openai/gpt-oss-20b` — второй не тестировался, MoE-архитектура gpt-oss иногда хуже дружит с grammar-constrained sampling, чем dense-модели, это стоит проверить отдельно перед включением на нём). Первая, самая безопасная точка внедрения — `intentLlmJudge.js` (уже возвращает одно слово из фиксированного списка, минимальный риск сломать что-то при переходе на `enum`-schema).
- **Разделение данных и инструкций для недоверенного OCR/PDF-текста** — внедрено сейчас (10.3.4), это буквально бесплатно (только текст промпта).
- **Semantic entropy / self-consistency** (сэмплировать ответ N раз, мерить согласованность/энтропию как сигнал неуверенности) — сознательно **не рекомендуется** для этого проекта: требует N дополнительных проходов модели на один запрос, то есть прямо противоречит требованию «не замедлять и не грузить общую T4». Подтверждается и внешними данными: standalone semantic entropy даёt AUROC ~0.60 на бенчмарках — сигнал слабый сам по себе, не окупает N-кратную стоимость для этого масштаба.
- **Dual-pass verification** (второй, отдельный проход LLM только как «grounded: true/false» верификатор, без права переписывать данные) — пользователь сам предложил это точно, и сам же верно указал не гонять на каждую позицию. Технически это то же самое, что уже есть в проекте как паттерн (`quoteIntentJudge.js`, теперь и `intentLlmJudge.js`) — узко заточенный LLM-судья, вызываемый только на пограничных случаях, а не на каждый запрос. Разница с §10.3.3 — тот судья про intent, а не про grounding конкретной позиции; отдельный grounding-верификатор для позиций с низким margin top-1/top-2 не реализован, это реалистичный кандидат на следующую итерацию, если появятся случаи `similar`/`analog` с высокой ценой.

**Явно не в приоритете (сознательно, по тем же причинам, что и §8.6):** Qwen3Guard-класс safety-классификаторов — не решают доменную задачу (не отличат DIN 931 от допустимого аналога DIN 933), это отдельная задача от grounding; выделенные узкие ShopDB-тулы вместо одного поиска — архитектурный рефакторинг, не точечный фикс.

### 10.5 Что проверить в проде

1. Повторить исходный репорт-диалог («Почему в КП нет цен?...») — ответ должен смениться на `ABSTAIN_MESSAGE` вместо выдуманного каталога.
2. Проверить `offerkp metrics` (§9) через 1-2 дня прод-трафика на предмет всплеска `matchType: "none"` — теперь абстинирование должно быть видно в метриках, а не молча появляться как «успешный» ответ с выдумкой.
3. Golden-фикстура `intentRouting.jsonl` не менялась (её ожидания по `edit_quote`/`allowShopDbSearch=true` уже были верны) — но стоит добавить в неё явный кейс на `bolt`/`screw` без иных сигналов, чтобы регресс по 10.3.2 не вернулся незаметно.

## 11. Второй раунд guardrails/grounding: приоритетный список техник + баг, найденный метаморфик-тестами (2026-07-21, дополнение 7)

Пользователь прислал приоритезированный список из 10 техник (error taxonomy, minimum information policy, metamorphic tests, disagreement routing, confidence calibration, quote invariant validator, shadow mode, property-based tests, temporal revalidation, closed generative surface) плюс отдельно — развёрнутый разбор подхода OpenAI к снижению галлюцинаций (tool-grounded facts, closed-world grounding, Structured Outputs, abstention states, evals с конкретными метриками типа `wrong_price_rate`, `false_exact_rate`, red-teaming на контрфактах типа M10x80→M10x70/DIN 933→DIN 931/болт→гайка). Задача — сверить со здравым смыслом/вебом и реализовать то, что дёшево и не тормозит.

### 11.1 Что реализовано в этот раз

**Metamorphic-тесты** (`__tests__/utils/offerKp/metamorphic.test.js`, новый файл) — ровно техника №3 из списка пользователя, нулевая цена в проде (только CI). Тестируют `classifyProductMatch` (`analogRules.js`) через два вида отношений, а не фиксированный оракул:
- **Invariance** — семантически нейтральные перефразировки (порядок слов, регистр, лишние пробелы, кириллическая «х» вместо латинской «x») не должны менять вердикт.
- **Sensitivity** — критичный признак (длина 80→70, диаметр M10→M8, класс прочности) обязан менять вердикт, а не «залипать» на `exact`.
- **Abstention** — отсутствие резьбы/стандарта в запросе обязано давать `size_unconfirmed`/не-`exact`, а не угадывание.

**Важный результат:** метаморфик-тесты **сразу нашли реальный P0-баг**, не гипотетический — см. §11.2.

**Error taxonomy** (техника №1) — `classifyProductMatch` уже давно считает подробную причину непопадания в exact (`mismatchReason`: `size_unconfirmed`, `spec_mismatch` с конкретным полем `product_type`/`coating`/`strength_class`), но эта причина никогда не доходила до непрерывного лога метрик (§9) — `recordSearchMetric` знал только `matchType`, не «почему». Добавлено поле `failureReason` в `matchInquiryLines.js` (переиспользует уже посчitанное значение, никакой новой работы) + разбивка «Причины непопадания в exact/analog» в `scripts/report-shopdb-metrics.cjs`/`offerkp metrics`. Теперь видно не только «57% none», а «из них 40% — не указан размер, 30% — не то покрытие, 20% — не тот тип изделия» — актуальная база для будущего hard-negative mining (§8.6) и калибровки (техника №5, всё ещё заблокирована отсутствием данных оператора).

### 11.2 Найденный и исправленный P0-баг: «гайка» матчилась как точный «болт»

При написании sensitivity-теста на product type (по образцу контрфактов из присланного разбора — «болт → гайка») выяснилось, что `classifyProductMatch("гайка DIN 933 M10x80", БолтDIN933)` возвращал **`exact`**, а не отказ — то есть система могла подтвердить ценой и SKU **гайку** для запроса, где каталожный товар на самом деле **болт**.

**Причина** — регресс от фикса из прошлой сессии этого же дня (`STANDARD_IMPLIES_TYPE` в `hardwareQuery.js`, добавленного для другой цели: чтобы колloquial «винт» вместо каталожного «болт» не исключался SQL-фильтром). Тот фикс аддитивно добавлял «болт» (тип, подразумеваемый DIN 933) в `parsed.productTypes` **поверх** любого явно названного клиентом типа. `productTypeMatches()` в `analogRules.js` проверяет совпадение по `.some()` (ИЛИ, не И) — значит если в списке оказались оба «гайка» и «болт», товар «Болт DIN 933...» проходил проверку типа, потому что «болт» там тоже был, хотя клиент явно попросил гайку.

**Исправление** (`hardwareQuery.js`) — implied-тип из `STANDARD_IMPLIES_TYPE` подставляется **только если клиент вообще не назвал тип** (`productTypes.length === 0`), а не поверх явно указанного. Проверено:

| Запрос | До фикса | После фикса |
|---|---|---|
| `DIN 933 M10x80` (тип не назван) | `болт` (implied) | `болт` (implied) — без изменений |
| `болт DIN 933 M10x80` | `exact` | `exact` — без изменений |
| `гайка DIN 933 M10x80` | **`exact`** (баг) | `spec_mismatch` / `product_type` |
| `винт DIN 933 M10x80` (исходный colloquial-кейс) | `exact` (через union) | `spec_mismatch` / `product_type` |

**Осознанный компромисс, не скрытый:** четвёртая строка — это фактический возврат прошлого бага (customer называет «винт» то, что в каталоге «болт», поиск снова не находит совпадение) ради устранения куда более серьёзного (называние **другой категории товара** точным совпадением с ценой). Между «система не нашла и написала "нет в базе"» и «система подтвердила гайку как болт с ценой» первое явно безопаснее — это то же самое предпочтение abstention над угадыванием, которое явно называет и присланный разбор OpenAI («лучше признать незнание, чем угадать»). Правильное окончательное решение — таблица допустимых colloquial-синонимов (винт≈болт можно простить, гайка≠болт нельзя), но это требует доменной проверки у оператора, которой нет — фабриковать её было бы тем же самым грехом, что описан в §6.4. Зафиксировано как открытый пункт в §11.4.

Регресс покрыт тестами: `metamorphic.test.js` (новый sensitivity-кейс на product type) + `hardwareQuery.test.js` (новый файл, 4 теста на `parseHardwareQuery.productTypes`). Полный прогон: **768/768** (было 753 до дополнения 6, 768 после дополнений 6+7), без единого regressions.

### 11.4 Открытые пункты после этого раунда

1. **Приоритет №1 (безопасность):** таблица допустимых colloquial-синонимов типа товара (винт≈болт можно, гайка≠болт нельзя) — нужна проверка с оператором каталога, не фабриковать самостоятельно.
2. Disagreement routing между lexical/semantic поиском — дизайн-задача, требует разделения результатов стратегий до слияния в один пул кандидатов.
3. Temporal price revalidation — сначала измерить частоту intraday-изменений цен в ShopDB (через уже существующий `searchMetrics.js`/прямой SQL), потом решать, нужен ли лишний запрос при экспорте.
4. Structured Outputs / grammar-constrained decoding (§10.4) — конкретный следующий шаг: `intentLlmJudge.js` как первая точка внедрения `response_format: json_schema`.
